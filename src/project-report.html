第一章 · 项目概览

项目定位

这是一套面向 SMT 产线/芯片烧录场景的 烧录座寿命智能监控系统。系统中每个烧录座有 1~4 个烧录头，设备通过特定格式的 .txt 文件上报 OK/NG/当前次数/累计次数。软件实时监控文件夹中文件的变化，统计每个烧录头的累计使用次数，结合预设的寿命阈值发出预警/警报，并支持报废管理、库存管理、维护记录等功能。

核心功能清单

监控 文件夹监控 .txt 文件变化
解析 4头 OK/NG/次数 自动解析
寿命 型号级寿命阈值 预警/警报
库存 Excel导入 型号+ID 库存管理
维护 探针/弹性/粉尘 维护检查记录
报废 自动扣减库存 报废记录追踪
主控 汇总所有分机数据 集中看板
分机 数据上报主控 接收主控指令
监控站 只读查看主控数据快照

第二章 · 整体架构与角色模式

三种运行角色

系统支持同一套代码以三种角色运行，通过界面 Setup 页配置切换：

主控 (Master)

启动 Express HTTP 服务（默认3456端口），接收所有分机上报数据，合并汇总统计。管理统一寿命配置、库存、维护记录。可向分机推送更新/删除指令。

核心：server.js + watcher.js 的 masterStatsMap

分机 (Client)

监控本地文件夹，数据变化后通过 HTTP POST 上报到主控。定时5秒心跳，接收主控回推的配置更新和删除指令。

核心：client.js 的 reportToMaster() + sendHeartbeat()

监控站 (Monitor)

只读模式，不监控本地文件夹。每3秒轮询主控的 /api/snapshot 获取完整数据快照，用于远程大屏展示。

核心：main.js 的 fetchMonitorSnapshot() + monitorPollTimer

// setup() 核心逻辑
setup() {
  // 状态响应式数据
  const role = ref(null)           // 当前角色: 'master'|'client'|'monitor'|null
  const connectionStatus = ref('未连接')

  // 监听主进程推送的 watcher:update 事件（完整数据包）
  const unsub = window.electronAPI.onWatcherUpdate((data) => {
    role.value = data.role             // 角色配置
    connectionStatus.value = data      // 含 serverInfo / clientStatuses
    // 播放报废通知声音
    if (data.scrapNotification) playSound('scrap')

  // 监听 connection:update 事件（2秒定时推送，仅连接信息）
  window.electronAPI.onConnectionUpdate((data) => {

  })

  onUnmounted(() => { unsubscribeAll() })
}

3.3 SetupPage.vue — 角色设置页

应用首次启动或清空配置后的入口页面。提供三种角色的一键配置：

3.4 DashboardPage.vue — 监控看板（核心页面）

这是系统最重要的数据展示页面，约700行代码，主要分为三个区域：

区域1：型号级寿命总览表

// 将寿命配置 (lifespanData) 与统计数据 (statRows) 按型号关联
// 每个型号一行，显示：
//   型号 | 寿命上限 | 在用设备数 | 最大使用次数 | 寿命百分比(进度条) | 状态标记
const adapterStats = computed(() => {
  lifespanData.value.map(item => {
    const devices = statRows.value.filter(s => s.model === item.adapter)
    // 取该型号下所有设备中累计次数最大的
    const maxUsage = Math.max(...devices.map(d => d.usageCount || 0), 0)
    const percent = (maxUsage / item.limit * 100)
    return { model: item.adapter, limit: item.limit, percent, status: ... }
  })
})
// 状态判定：>=100% 超寿命(红) | >=alertCount 警报(橙) | >=warnCount 预警(黄) |

区域2：设备级详细统计表（带折叠分组）

按型号分组，每个型号内展开显示每台设备(型号+机器ID)的每个烧录头独立统计：

  • 头1~4 的累计使用次数（从 cumulativeCount 数组取）

  • NG报警：当该头 NG 计数 > ngAlertCount 时标记

第四章 · 核心数据流与联动关系

4.1 数据采集链路

烧录设备
→
写入 .txt 文件
→
Chokidar 监听 add/change/unlink
→
parseContent() 解析4组数据
→
updateStats() 按头独立统计
→
pushUpdate() 推送到渲染进程

4.2 数据格式详解

文件名: 型号_机器ID$任意.txt

文件内容 (以制表符分隔4组，每组用冒号分隔4个头):

OK组:  100:50:200:0     ← 头1 OK=100, 头2 OK=50, 头3 OK=200, 头4 OK=0
NG组:  2:0:5:0          ← 头1 NG=2,  头2 NG=0,  头3 NG=5,   头4 NG=0
当前组: 50:30:80:0       ← 当前工单各头已烧录次数
累计组: 5000:3200:8000:0 ← 此行最重要！各头累计总次数 = 寿命消耗

关键：解析后每个头独立创建一条 stat 记录（key=型号|机器ID|头索引），usageCount 取自累计次数组对应头位置的值。在 Dashboard 中每个设备会展开显示1~4行（按实际活跃头数）。

自动恢复 (autoRestore)：如果 app-config.json 中存在角色配置，启动时自动恢复： 主控→startServer / 分机→startHeartbeat+report / 监控→startMonitorPolling。这意味着产线断电重启后无需人工重新配置。

5.2 watcher.js — 文件监控核心（~800行）

核心数据结构：

// dataMap: 文件路径 → 解析后的文件记录
Map<filePath, { fileName, content, model, machineId, headCount, ok[], ng[], currentCount[], cumulativeCount[], ... }>

// statsMap: "型号|设备ID|头索引" → 统计记录（按头分开）
Map<key, { model, machineId, headIdx, count, usageCount, updatedAt, deleted, hidden }>

// masterStatsMap: "型号|设备ID|分机ID" → 分机统计
// 仅主控模式使用，用于合并所有分机数据
Map<key, { ...stat + clientId, clientName }>

文件名解析 (parseFileName)：型号_机器ID$任意.txt → 取第一个 _ 之前的为型号，_ 到 $ 之间为机器ID。

内容解析 (parseContent)：按 \t (制表符) 分割4大组，每组按 : 分割4个头 → 分别得到 OK/NG/当前/累计的四头数组 → usageCount = cumulativeCount[headIdx]。

状态判定： hidden（手动隐藏）> deleted（异常：文件消失）> standby（5分钟无更新）> active（正常活跃）。

主控汇总 (getMasterStats)：合并本机 statsMap + 分机 masterStatsMap → 同型号同机器同头合并 → count累加 → usageCount取最大值。

统一监控数据 (getAllMonitorData)：返回本机 + 所有分机的 records 和 stats 统一数组，每行标注 source（本机/分机名）和 dataType（record/stat），供管理后台使用。

主控指令处理： applyUpdatesFromMaster() — 分机接收并执行主控的更新指令（维护/寿命/库存/stat_update/record_update）； deleteLocalDataByCommand() — 分机接收删除指令。

5.3 server.js — 主控 HTTP 服务（~300行）

基于 Express 的 REST API 服务：

端点方法说明
/api/reportPOST分机上报统计数据。接收 {clientId, clientName, records, stats}，更新心跳记录，回调 main.js 的 serverCallback
/api/heartbeatPOST分机心跳（5秒间隔）。返回 {needResync, deleteCommands, updateCommands}。主控通过此通道下发更新/删除指令
/api/pingGET分机测试连接用
/api/maintenance-reportPOST分机上报维护变更（报废/全OK/NG抑制）。触发 onMaintenanceReport → 同步主控 + 广播其他分机
/api/snapshotGET监控模式数据快照。调用 snapshotProvider() 返回全部数据

指令队列：clientDeleteQueues / clientUpdateQueues — 主控修改数据后不主动推送，而是放入队列，分机下次心跳时一并拉取。
去重：同类型同设备的更新只保留最新一条。
超时清理：30秒无心跳的分机自动从 clientStatus 移除。

5.4 client.js — 分机上报模块（~180行）

分机作为 HTTP 客户端，向主控上报数据：

  • reportToMaster(data)：POST /api/report，发送 {clientId, clientName, records, stats}，失败自动重试最多3次（间隔1秒）
  • sendHeartbeat()：POST /api/heartbeat，检查响应中的 needResync → 触发重新上报；处理 updateCommands/deleteCommands → 调用回调更新本地
  • startHeartbeat()：每5秒发一次心跳
  • testConnection(host, port)：GET /api/ping 测试主控可达性
  • reportMaintenanceUpdate()：POST /api/maintenance-report 上报维护变更
  • 回调：setResyncCallback / setDeleteCallback / setUpdateCallback 用于外部注入处理逻辑

5.5 config-store.js — 配置持久化

单文件 JSON 存储（app-config.json），结构：

{
  role: null | 'master' | 'client' | 'monitor',
  serverPort: 3456,       // 主控服务端口

第六章 · Store 数据模块详解

6.1 lifespan-store.js — 寿命配置管理

存储各型号烧录座的寿命阈值，位于 machine-lifespan.json：

每条记录: {
  adapter: "AT89S52",        // 型号名
  limit: 30000,              // 寿命上限（累计次数超过此值即超寿命）
  warnCount: 25000,          // 预警阈值 = limit - 5000
  alertCount: 28000,         // 警报阈值 = limit - 2000
  ngAlertCount: 0           // NG报警阈值（测试中出现NG的次数）
}

Excel 导入：列名 Adapter / Limit → 新增机型（去重）、已有机型如寿命不同则更新并重新计算默认阈值。
广播：主控更新后自动广播到所有分机。

6.2 socket-inventory-store.js — 烧录座库存管理

两个文件：socket-inventory.json（库存条目）+ socket-inventory-config.json（型号级配置）：

// 库存条目：每个型号+ID 一条
{ model: "AT89S52", machineId: "K001", stock: 5 }

// 型号级配置：预警水位、使用数量
{ "AT89S52": { warnLevel: 5, usedQty: 2, stockQty: 10, totalQty: 12 } }

核心功能：

  • Excel 导入：解析合并单元格（型号级汇总 + 逐行ID），自动关联
  • decrementStock()：报废时自动 -1，库存为0则删除条目，返回型号剩余库存
  • getGrouped()：按型号分组聚合，供前端表格展示
  • 主控更新后自动广播到所有分机

6.3 socket-maintenance-store.js — 维护记录

位于 maintenance-data.json，每条设备维护记录：

{
  model, machineId,       // 设备标识
  pin: 'OK'|'NG'|null,      // 探针检查结果
  elasticity: 'OK'|'NG'|null, // 弹性检查结果
  dust: 'OK'|'NG'|null,      // 粉尘检查结果
  maintenanceCount: 0,      // 维护次数（全部OK时才+1）
  lastMaintenanceTime: null, // 最后维护时间
  scrapTime: null,          // 报废时间（不为null表示已报废）
  ngAlarmSuppressed: false, // NG报警已抑制
  clientId: null            // 数据来源分机
}

自动联动：三项检查全部OK → 自动抑制NG报警；任一NG → 标记报废 + 扣库存。
分机互通：分机报废自动上报主控 → 主控同步 + 广播给其他分机（排除上报分机）。

6.4 master-device-store.js — 主控设备表

位于 master-device-db.json，仅主控模式使用。记录分机上报的原始 records 数据：

每条记录: {
  model, machineId,        // 设备标识
  okTotal, currentCount, cumulativeCount,
  headCount,
  ok: [n,n,n,n], ng: [n,n,n,n],      // 四头数组
  currentArr: [n,n,n,n], cumulativeArr: [n,n,n,n],
  updatedAt, clientId, clientName
}

key = 型号|机器ID|分机ID。用于管理后台查看原始分机数据和按型号合并展示（getAll() 会自动 merge 同型号同ID不同分机的数据）。

6.5 scrap-records-store.js — 报废记录表

独立记录所有报废操作历史，位于 scrap-records.json：

{ model, machineId, scrapTime, cumulativeCount, source, operator }

每次 markScrap / recordMaintenance(全NG) / 分机上报报废时自动追加。前端通过 scrapData 字段在 Dashboard 中展示。

总结

架构亮点

  1. 三角色同构：同一套代码通过配置切换主控/分机/监控三种角色，减少维护成本
  2. 最终一致性：主控→分机采用心跳拉取而非实时推送，产线网络不稳定时也能可靠同步
  3. 按头独立统计：1个烧录座=1~4个头，每个头独立监控使用次数，精确到每个烧录头的寿命管理
  4. 自动恢复：断电重启后自动恢复角色和文件监控，无需人工干预
  5. 完善的崩溃处理：多层异常捕获（主进程/渲染进程/GPU）+ 运行标记检测异常退出
  6. 32位兼容：针对32位Windows系统禁用GPU加速避免崩溃
  7. JSON文件存储：无需额外数据库依赖，所有数据持久化在 userData 目录的JSON文件中，方便备份和迁移
  8. 库存自动联动：报废自动扣库存 + 广播通知 + Toast消息 + 声音提示
